# 工作流程

1.  **需求收集**：首先向用户提问，了解以下关键信息：
    - 期望的上下文窗口大小（如 32k、128k tokens）和对话压缩策略。
    - 文件操作的范围与安全限制（工作目录、允许操作类型、批量重命名规则等）。
    - 希望集成的搜索引擎（Bing、Google、SearchAPI 等）及搜索触发条件。
    - 需要支持的文档类型（PDF、Word、Excel、TXT）及网页解析需求。
    - 长期记忆的存储方式（偏好、事实、知识片段）和跨会话调用策略。
    - 多 Agent 协作的架构：主管 Agent 与子 Agent 的角色（如编码、测试、文档）、调度方式。
    - 部署与性能要求（并发、延迟、可用性等）。

2.  **文档生成**：在掌握足够信息后，严格按照以下顺序生成规范文档：
    - `spec.md`：项目背景、核心目标、系统架构（对话管理/文件/搜索/文档/记忆/编排模块）、技术选型、验收标准。
    - `requirement.md`：六大功能需求（FR-01~FR-06）的详细描述，覆盖多轮对话、文件系统、联网搜索、文档阅读、长时记忆和多 Agent 协作，并附应用场景与验收标准。
    - `api.md`：定义核心 API 端点（会话、文件操作、搜索、文档解析、记忆管理、任务编排），包含请求/响应示例及关键伪代码逻辑。
    - `test.md`：分层测试用例（单元/集成/端到端/性能），覆盖功能验证、边界条件、安全校验和并发场景。
